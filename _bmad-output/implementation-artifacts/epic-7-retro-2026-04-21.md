## What Went Well

1. **Playwright E2E infrastructure shipped cleanly in Story 7.1**  
   From zero to a working E2E suite with `pnpm run check:e2e` in one story. The `webServer` autostart pattern, Chromium-only target, and `reuseExistingServer: !CI` flag were exactly right. No debug issues during implementation — the dev notes guided a direct path to passing tests. Four review patches strengthened the coverage (dual error capture, scoped nav locators, exact URL matching, brand link coverage).

2. **Story 7.5 delivered the most comprehensive test suite in the project's history**  
   41 tests covering workflow actions, artifact consistency, and session log states. The network-mocking pattern with `page.route()` and fixture files in `tests/fixtures/` makes tests deterministic — no live dev server dependency for API responses. The shared `mockApi` helper in `tests/helpers/mock-api.ts` reduces boilerplate and is the right abstraction. The EventSource SSE mock via `abortRoute` was a clean solution to prevent connection noise.

3. **Story 7.3's self-referential validation proved Phase 1 coherence**  
   The traceability loop is real and demonstrable: `epics.md → story spec files → Copilot CLI sessions → agents-sessions.json → bmad-ui views`. All seven core routes (home, workflow, epic detail, story detail, sessions list, session detail, analytics) render correctly with real project data. Zero rendering bugs found during validation. The `docs/phase-1-completion.md` document gives Phase 2 a clear baseline.

4. **TanStack Query migration in Story 7.2 addressed a pre-existing code quality violation**  
   `epic.$epicId.tsx` was using `useEffect + useState` for data fetching — explicitly forbidden by project rules. Story 7.2 replaced this with two `useQuery` calls with proper `refetchInterval` and `staleWhileRevalidate` behavior. This is the pattern all future routes should follow. The planned-only story rendering (dimmed rows for stories not yet in sprint-status) is a product feature that emerged cleanly from the architecture change.

5. **Story 7.4 executed a valuable artifact audit**  
   `project-context.md` was corrected: aspirational Phase 2 rules (shadcn/ui, Base UI, `src/ui/` hierarchy) are now clearly marked as `[Phase 2 planned]` so AI agents don't implement against a phantom stack. The architecture.md `Phase 1 Delivered` section gives future contributors accurate historical context. These corrections were overdue.

6. **`pnpm check` discipline held across all five stories**  
   Every story verified `pnpm check` (lint + types + tests + build) passing before marking done. No regressions to the existing quality gate were introduced. This is the sixth consecutive epic with this discipline intact.

## Challenges and Growth Areas

1. **Same-commit rule broken in Stories 7.2 and 7.4 — the rule we've enforced for two epics**  
   Stories 7.2 and 7.4 show `Status: in-progress` in their story files while `sprint-status.yaml` marks them `done`. This is precisely the inconsistency the same-commit rule was designed to prevent. We've enforced this rule cleanly in Epics 5 and 6. It regressed in Epic 7.  
   
   _Root cause: Both stories closed via sprint-status update without a corresponding story file status update. The commit that changes sprint-status must also update the story file._

2. **Unresolved review findings create latent product bugs**  
   Story 7.2 deferred two review findings: planned-only rows are appended unsorted (story order doesn't follow ticket order), and `Plan all stories` includes already-created stories (could cause redundant create-story runs). Story 7.4 deferred two: the Phase 1 completion claim contradicts sprint tracker state, and the Phase 2 deferred list is incomplete. These are not cosmetic — the plan-all-stories bug is a functional correctness issue. They need story files, not checkbox entries.

3. **Epic 8 stories partially duplicate Epic 7 work — a significant discovery requiring re-planning**  
   Epic 8's planned stories include:
   - Story 8.2: "Integrate Backlog Artifacts with UI Workflows" — same title as Story 7.2 (already done)
   - Story 8.3: "Validate Self-Referential Delivery Loop" — same title as Story 7.3 (already done)
   - Story 8.4: "Establish Phase Boundary and Evolution Guardrails" — same title as Story 7.4 (already done)
   
   Stories 8.2, 8.3, and 8.4 are ready-for-dev but their acceptance criteria overlap with what Epic 7 delivered. Before executing Epic 8, these stories need review to determine whether they are superseded by Epic 7's work or whether they define genuinely new scope.

4. **Node.js 22 JSON import assertion compatibility surfaced unexpectedly in Story 7.5**  
   Test files required `with { type: "json" }` import assertions for JSON fixture imports — not documented and not expected. Node 22's stricter import assertion handling broke the initial implementation. The fix was straightforward, but the issue highlights that the test environment has Node.js version constraints not fully documented in dev notes.

5. **Epic 7 scope was a hybrid that didn't match its title**  
   The epic name is "End-to-End Testing & Regression Safety" but only Stories 7.1 and 7.5 delivered E2E testing. Stories 7.2–7.4 delivered product features, validation, and documentation. This made the epic difficult to reason about as a unit and contributed to story scope uncertainty during execution.

## Key Insights and Learnings

1. **Story files are the only reliable enforcement mechanism — action item tables don't survive sprint transitions**  
   Epic 7 resolved exactly one item from Epic 6's action table: Story 5.1, which was already a story file. Four of seven remaining action items went unaddressed. This is the fourth consecutive retrospective to make this observation. The mechanism to break the cycle is simple and known: any deferred item that appears in a retro table for a second time must become a story file before the next epic starts.

2. **Network mocking is the right architectural pattern for UI tests in this stack**  
   Story 7.5 proved that `page.route()` with fixture JSON delivers deterministic, fast, and environment-independent tests. The fixture files in `tests/fixtures/` are documentation as well as test data — they define the exact API contract expected by each view. This pattern should be the standard for all future E2E tests.

3. **The self-referential loop is real and operational**  
   bmad-ui was built using bmad workflows. bmad-ui can observe that fact through its own interface. Story 7.3 demonstrated this is not theoretical: 46 stories, 234 sessions, and all core views render correctly from real project data. Phase 1 delivered its foundational promise.

4. **Aspirational rules in project-context.md confuse AI agents**  
   Before Story 7.4's audit, `project-context.md` described `src/ui/` directories, shadcn/ui components, and Base UI primitives as if they existed. They don't. AI agents implementing new features would have referenced a phantom stack. The `[Phase 2 planned]` annotation is the right convention — it preserves forward intent without misleading current implementation.

5. **Review findings deferred as unchecked checkboxes become invisible technical debt**  
   Four review findings across two stories (7.2 and 7.4) were deferred with unchecked checkboxes. Without corresponding story files, these findings have no implementation vehicle. The pattern is clear: any deferred review finding must become a story file entry in sprint-status.yaml, not a checkbox that persists in the story file indefinitely.
