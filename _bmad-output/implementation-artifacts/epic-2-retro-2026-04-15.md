## What Went Well

1. **Surgical Terraform extension in Story 2-1** — Clean extension of existing skeleton with 9 in-place updates, no resource recreation, no drift. `terraform apply` worked as expected.

2. **Well-architected 4-job pipeline in Story 2-2** — `check-changes`, `infra-deploy`, `deploy-preview`, `deploy-production` jobs are clearly separated. Intelligent skip logic prevents unnecessary runs.

3. **Fast iteration on CI failure** — First CI run failed on pnpm setup (missing `package_json_file`). Fixed and re-run in the same session. Healthy, non-spiraling debugging.

4. **Elegant secrets model** — Single `DOTENV_PRIVATE_KEY` GitHub Secret unlocks all downstream secrets via dotenvx. Simple to reason about, simple to audit.

5. **Proactive AC validation** — Story 2-3's dev notes correctly identified that the Terraform environment separation AC was invalid (single repo, no split needed) and documented it explicitly.

---

## Challenges and Growth Areas

1. **Credential bootstrap friction appeared in both stories**
   - Story 2-1: `terraform plan/apply` blocked until `.env` with `GH_PAT_TOKEN` and `GITHUB_OWNER` was available
   - Story 2-2: Completed with `VERCEL_TOKEN`/`VERCEL_ORG_ID` still absent from `.env` — deploy-preview and deploy-production jobs have never run end-to-end
   - Pattern: stories were marked done before their deployment paths were fully exercised

2. **Monorepo `package_json_file` friction in CI**
   - `pnpm/action-setup@v5` requires explicit `package_json_file` when `package.json` is not at repo root
   - This pattern will recur in every new GitHub Actions workflow unless documented

3. **AC became obsolete before dev started on Story 2-3**
   - The original AC1 (Terraform env separation) was invalidated by the project's single-repo structure
   - Dev notes documented this — but the story was still scoped and created with the invalid AC
   - Leads to cancelled/descoped work rather than properly pre-validated stories

4. **Story 2-3 scope partially absorbed without explicit handoff**
   - Parts of 2-3 (workflow job separation, environment gate) were implemented in 2-2
   - The deployment runbook AC was deferred — `docs/deployment-guide.md` remains a stub

---

## Key Insights and Learnings

1. **Credential availability is a story exit criterion** — A story that delivers infrastructure which requires secrets should verify those secrets are populated before marking "done."

2. **Monorepo CI setup has non-obvious friction points** — Document the `package_json_file` pattern and other pnpm workspace gotchas in a project-level reference.

3. **Story ACs should be validated for obsolescence before dev starts** — Not just at acceptance. Prevents cancelled scopes and wasted spec-writing effort.

4. **Story 2-3's cancellation was net-positive** — It avoided implementing Terraform workspace complexity for no real benefit. The cancellation was the right call.

5. **pnpm check passes ≠ deploy pipeline is functional** — Healthy CI check-changes run can mask incomplete deployment infrastructure.
