## What Went Well

1. **100% story completion with clean execution**  
   All 7 stories delivered as planned. No stories were cancelled, deferred, or significantly descoped. Epic 9 is the most consistent delivery record in the project's recent history.

2. **`loading-states.tsx` was created once and reused — the right pattern**  
   Story 9.1 created `src/lib/loading-states.tsx` with `PageSkeleton`, `EmptyState`, and `ErrorState` components. Story 9.6 reused `EmptyState` for the "No active sessions" panel. This is exactly the component extraction model the project-context.md specifies — build it when you need it, in `src/lib/`, then reuse.

3. **Design system compliance was near-perfect across all stories**  
   Every story's dev notes explicitly called out `var(--muted)`, `var(--highlight)`, `var(--panel)` usage. No hardcoded hex colors were introduced. CSS variable discipline has clearly become a team habit in this epic.

4. **`StatusBadge` (Story 9.3) eliminated an entire class of inconsistency**  
   Before Story 9.3, badge rendering was scattered across multiple components with one-off inline styles. The refactor unified all status rendering through a single `StatusBadge` component backed by the canonical `.step-badge` CSS variants. This is a quality multiplier — future stories automatically get consistent badges without effort.

5. **Story 9.7 delivered a complex feature with excellent architectural discipline**  
   Step detail views required new routes, a new `/api/workflow-step-detail` endpoint, a new static data emitter in `vite-plugin-static-data.ts`, and dual-mode (dev/prod) support — all without violating the existing architecture contract. The dev notes were detailed and accurate. The implementation followed the established `apiUrl()` + TanStack Query pattern throughout.

6. **Review process caught real issues in every story**  
   No story sailed through without review patches. The review pass on 9.6 caught a loading state gap, a click-event forwarding bug, and inline-style violations — all fixed in-sprint. This is the retrospective process paying dividends: stories are now reviewed seriously.

7. **TanStack Router `activeProps` in Story 9.2 fixed a silent correctness bug**  
   Manual `aria-current` comparison was failing on nested routes. Switching to `activeProps` with `isActive` automatically handles route hierarchy — a permanent fix rather than a workaround.

## What Didn't Go Well

1. **Story 9.5 violated the same-commit rule — again**  
   Story 9.5 (Session Detail Back Navigation) is marked `done` in sprint-status.yaml but has no story file in `_bmad-output/implementation-artifacts/`. This is the exact violation the Epic 7 retro identified as a recurring problem. We fixed 7.2 and 7.4, then broke the rule with 9.5. The rule states: any change to sprint-status must include a corresponding story file status update in the same commit.

2. **TanStack Query migration carries forward into Epic 10**  
   `story.$storyId.tsx`, `session.$sessionId.tsx`, `prepare-story.$storyId.tsx`, and `analytics-utils.tsx` still use `useEffect` for data fetching. This was documented in Epic 7's retro as Phase 2 debt. Stories 8.3 and 9.1 both explicitly call it out as "out of scope." It is now genuinely blocking — Epic 10 stories that touch these routes will encounter the anti-pattern immediately.

3. **Epic 4 CI/CD patches are six epics old — this is now critical technical debt on live workflows**  
   Seven review patches on `.github/workflows/` files were deferred after Epic 4. They apply to CI/CD workflows running in production. They have survived every retrospective since Epic 4 as a table entry. The principle from Epic 7's retro ("any item that appears in the retro table for a second time must become a story file") has been violated four times.

4. **Story 9.3 deferred three agent-server.ts findings — pre-existing but accumulating**  
   Three review findings were deferred as "pre-existing" in Story 9.3:
   - Brittle markdown status regex (`agent-server.ts:384`)
   - Markdown sync logic preventing explicit status resets (`agent-server.ts:1284`)
   - Silent error suppression on implementation-artifacts reads (`agent-server.ts:1328`)
   
   These are production correctness issues in the data ingestion layer. They were correctly deferred out of a polish story's scope, but they need story files.

5. **Story 9.4 had scope drift — StatusBadge refactor shipped in a layout story**  
   The review findings note that a `StatusBadge` refactor was included in a commit for Story 9.4 (a CSS-only layout story). The refactor was the correct direction but introduced untargeted scope. The review deferred it as pre-existing, but drift like this makes commit history ambiguous.

6. **The dotenvx auto-mask caveat is nine epics old and still not in project-context.md**  
   This is a one-line documentation addition that prevents future AI agents from accidentally exposing secrets in logs. It has been in every retrospective since Epic 1. Nine-epic carry on a documentation fix is a process failure.

## Key Insights and Learnings

1. **Table entries don't get done. Story files do.**  
   For the fifth consecutive retrospective, we observe that action items without story files are not executed. Story 9.5 itself is a symptom: it was executed (the feature is shipped) but never formalized into a story file. The enforcement mechanism must change from retro table → story file creation as a blocking pre-condition for starting the next epic.

2. **The `src/lib/` pattern for shared components works — commit to it**  
   `loading-states.tsx` proved the hypothesis. Shared UI utilities belong in `src/lib/`. This should be the explicit home for any cross-cutting React component that is reused across 2+ routes. The project-context.md already states this; the team followed it cleanly in Epic 9.

3. **Design system compliance is now a habit, not a check**  
   No story in Epic 9 was flagged for hardcoded colors or wrong CSS class usage in the initial implementation. The design system rules have been internalized. This is the long-term payoff from the enforcement in earlier epics.

4. **Story 9.7's complexity was managed by excellent dev notes**  
   The story file specified exactly which files to touch, the dual-mode contract to follow, and the edge case to handle (UX artifact missing). When a story is complex, up-front specification in the dev notes prevents scope creep and reviewer surprises. This pattern should be required for any story touching more than 3 files or introducing new routes.

5. **The same-commit rule needs automated enforcement**  
   Three violations across Epics 7, 8, and 9 (7.2, 7.4, 9.5) all share the same pattern: developer marks sprint-status done, forgets or skips the story file update. A pre-commit hook or `pnpm check` rule that validates story file `Status:` matches sprint-status entries would eliminate this class of violation permanently.
