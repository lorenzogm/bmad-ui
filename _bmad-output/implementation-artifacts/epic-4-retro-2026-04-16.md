## What Went Well

1. **Full CI/CD pipeline operational end-to-end — the epic's core goal**
   `bmad-ui-ci` runs on every PR, enforces Lint → Type check → Tests → Build. `bmad-ui-deploy` triggers on push to `main` with working Vercel deployment. Branch protection now enforces `bmad-ui-ci / Validate` as a required check. The complete automated loop from PR to production exists and works.

2. **Branch protection was toothless before — now it has teeth**
   Pre-Epic 4, `contexts = []` in Terraform meant branch protection existed but enforced nothing specific. Story 4.2 changed that with a precise context string (`bmad-ui-ci / Validate`) discovered from the GitHub Actions Jobs API. Merge-blocking is now real.

3. **Story 4.2 context string discovery was handled properly**
   Rather than guessing the context format, the story's dev notes prescribed the exact discovery method (run workflow, check Jobs API for registered context). This meant the Terraform change was precise on first application — no trial and error.

4. **Individual step naming across CI and deploy workflows enables precise diagnosis**
   Before Epic 4, a failing `pnpm check` in deploy.yml produced one opaque failure. After Stories 4.1, 4.3, 4.4: every failure names exactly which gate failed. Story 4.5 was itself a validation of this — diagnosing 9 files worth of Biome violations was straightforward because the CI step output was clean.

5. **Story 4.5's biome auto-fix was a one-shot unblock**
   `biome check --write` auto-fixed all formatting violations across 9 files. Only 3 items required manual fixes (unused variable, useExhaustiveDependencies, unused suppression). This demonstrates that the Biome tooling investment pays dividends — violations are fixable in bulk.

6. **Story sequencing was clean — no scope overlap**
   4.1 created ci.yml → 4.2 used its context string → 4.3 modified deploy.yml (separate file) → 4.4 added summaries to both → 4.5 validated the result. No two stories touched the same file at the same time. Dependencies were respected.

## Challenges and Growth Areas

1. **Story 4.5 emerged as an unplanned story — CI was broken on `main` from day one of Epic 4**
   Biome violations introduced by pre-Epic 4 feature work (analytics ECharts, sprint-status fix) meant the CI workflow that Story 4.1 created couldn't pass on `main` from its first run. Epic 4 started with a broken pipeline. Story 4.5 fixed this but it wasn't in the original sprint plan, which is why it never made it into sprint-status.yaml.
   
   _Root cause: no pre-epic CI check ran against the baseline before stories started. The new workflow exposed existing debt immediately._

2. **7 review patches across 3 stories remain open**
   Stories 4.1 (1 patch), 4.2 (2 patches), 4.4 (4 patches) each have open `[Review][Patch]` items in their files. None were applied as part of the story execution. This is a different pattern from previous epics where review patches were part of story completion. The review patches represent real quality gaps — CI summary missing pass/fail status, deploy summaries missing branch context, inject-safe writing concerns.

3. **deploy.yml `infra-deploy` scope gap was a hidden workflow assumption**
   Story 4.2's completion notes reveal: _"deploy.yml's infra-deploy job manages infra/vercel/ only; GitHub branch protection applied via local terraform apply."_ This means any GitHub infrastructure change requires knowing to run terraform locally — the deploy workflow only manages Vercel infra. This is undocumented and is a workflow trap for future maintainers.

4. **Epic 3 action items (3 of 4) carried forward again**
   This is now a pattern across three consecutive epics. Deferred action items consistently fail to complete unless they become story tasks. The team agreed in the Epic 3 retro that "action items without story files will not survive the next epic transition." That rule was not enforced.

5. **Story 4.5 not added to sprint-status.yaml**
   An emergent story completed the epic but was never formally tracked. sprint-status.yaml still shows `epic-4: in-progress`. This creates visible inconsistency between the implementation artifacts and the tracking system.

## Key Insights and Learnings

1. **The first CI run is a debt audit** — When you add CI to an existing codebase, the first run reveals existing violations. Epic 4 should have budgeted a "clean the baseline" story at the start (or validated cleanliness before starting). Story 4.5 emerged because this wasn't done proactively.

2. **Review patches are not optional cosmetics** — The 4 open patches in Story 4.4 include: missing CI pass/fail status in summary header, missing branch/ref context, `deploy-production` summary step skipped on failure (`if: always()` missing), and potential markdown injection edge cases in summary output. These are AC-adjacent gaps that should have been resolved during story execution, not left open.

3. **deploy.yml Terraform scope is an undocumented workflow boundary** — `infra-deploy` job = Vercel only. GitHub infra = local terraform apply. Any maintainer who doesn't know this will be confused when a GitHub infra change in a PR doesn't auto-apply. This needs documentation in `infra/github/README.md` or CONTRIBUTING.md.

4. **Emergent stories need immediate sprint-status registration** — Story 4.5 exists, was completed, but isn't tracked. This creates phantom gaps in the project record. The rule: if you create a story file, it must be added to sprint-status.yaml within the same session.

5. **Three-epic carry on deployment-guide.md is now a reliability signal** — An item that has been an "action item" for three consecutive retrospectives without becoming a story task is effectively never happening. It needs to either become Story 5.X with a proper AC, or be formally accepted as permanent documentation debt with an explanation.

### Key Technical Considerations for Epic 5

1. **npx install CLI scope** — Story 5.1 creates a CLI that copies `_bmad-ui` into a target bmad project. The self-contained nature of the app (all deps in `_bmad-ui/package.json`) is the enabling property. Epic 4 validated this works cleanly.

2. **pnpm → npm consideration** — The epic description says "using npm" for portability. But the current app is pnpm-based and CI enforces `--frozen-lockfile`. If the install CLI switches the app to npm, CI workflows and package.json scripts need updates. This is the single biggest scope ambiguity.

3. **No CI workflow changes needed for Epic 5 content** — Epic 4's CI pipeline is correct as-is. Epic 5 stories modify source files and package configuration, not workflows.

4. **sprint-status.yaml entries** — Stories 5-1, 5-2, 5-3 are already in sprint-status. No tracking changes needed at epic start.
