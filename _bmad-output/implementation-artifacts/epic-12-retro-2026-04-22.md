## What Went Well

1. **The structural work was scoped precisely and executed without drift.**
   Epic 12 had a clear, concrete problem statement — nested path confusion, scattered data files, broken CLI — and all three stories delivered exactly what was planned. No scope creep, no partial implementations. `pnpm check` passes, the directory layout is clean.

2. **The mass rename (12-1) was surgical — ~100 files, zero broken references.**
   The rename from `_bmad-custom/bmad-ui/` → `_bmad-ui/` touched GitHub Actions workflows, Terraform configs, documentation, VS Code settings, Copilot instructions, story files, and server code. Every reference was updated in a single commit. The follow-up commit (`0ed7e3c`) that re-added `scripts/server/logs/` to git was a minor miss caught and fixed in the same session.

3. **`artifacts/` consolidation creates a permanent, clear organizational separation.**
   `links.yaml`, `notes.json`, `story-dependencies.yaml` now live in `_bmad-ui/artifacts/`. `notes.yaml` (redundant duplicate) was deleted. The separation between runtime data (`agents/`) and project planning data (`artifacts/`) is now explicit and correct. Future contributors will not confuse them.

4. **Version 0.2.0 shipped to npm via GitHub Actions.**
   The CLI fix was not just internal — it was published. New bmad users can now run `npx bmad-method-ui install` and get a working install. That's a meaningful external-facing outcome from a 3-story structural sprint.

5. **High execution velocity on a wide-scope epic.**
   Structural epics involving mass renames often stall or require multiple passes. Completing all three stories in a single session with clean git history demonstrates strong preparation and focus.

   Epic 12 executed 3 stories. None produced story files in `_bmad-output/implementation-artifacts/`. 2. **Story 9.5 story file still missing — two retros after it was flagged.**
   Epic 9's retro called this "Action Item 1 — Before Epic 10 Story 1." Epics 10, 11, and 12 have since completed. The story file still does not exist.

3. **Epic 4 CI/CD patches are a nine-epic carry on live production workflows.**
   Seven review patches on `.github/workflows/` have survived every retrospective from Epic 4 through Epic 12. These are patches to CI/CD workflows actively running on `main`. The Epic 9 retro called this "Critical." Nothing changed across the three subsequent epics.

   One line of documentation to prevent AI agents from leaking secrets in logs has appeared in every retrospective since Epic 1. It is the longest-running unresolved item in this project's history.

6. **`npm publish` not verifiable in commit history.**
   Story 12.3's acceptance criteria explicitly required `npm publish` with version confirmation. The publish happened via GitHub Actions, but there is no `release:`, `chore:`, or tag commit that makes this traceable in the git log.

## Key Insights and Learnings

1. **The same-commit rule has no mechanical enforcement — and that's the root cause.**
   Five retros have documented same-commit violations. The Epic 9 retro recommended a pre-commit hook. The hook was never built. Until `pnpm check` or a git hook *fails* when a story is marked done without a corresponding story file, the rule will continue to be violated under execution pressure.

2. **Retro action items are subject to the same problem as all other action items: no story file = no execution.**
   This retro's meta-lesson: the Epic 9 action items were never converted into story files. They remained as table entries in a markdown document. Predictably, they were not done. Every retro action item that needs to be executed must become a story file with `ready-for-dev` status immediately after the retro.

3. **Structural epics have lower visibility — and therefore higher story-file risk.**
   Story 12-1 was a large, high-impact change that a future contributor will absolutely want to understand. "Why did the paths change?" Without a story file, they have only the commit message and the epic definition. For structural work, story files are *more* valuable, not less — because the decisions are less obvious than in feature work.

4. **The nine-epic carry on Epic 4 CI/CD patches is now a project risk, not just debt.**
   These are patches to production CI/CD workflows. If those workflows have subtle bugs, every PR, deploy, and publish has been running with known issues for nine epics. This must become a story file and be resolved in Epic 13 or as a pre-epic task.
