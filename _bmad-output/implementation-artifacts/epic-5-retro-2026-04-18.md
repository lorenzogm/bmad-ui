## What Went Well

1. **Story 5.2 was an efficient verification sprint — tooling was already correct**  
   Biome, TypeScript, CI, and `pnpm check` were all configured correctly before Story 5.2 started. The story's value was confirming that baseline and adding the Developer section to README. Zero regressions; clean exit. This is a good template for "verify and document" stories.

2. **Story 5.3 delivered a complete VS Code setup with correct formatter priority**  
   Creating `.vscode/extensions.json` and extending `.vscode/settings.json` with per-language Biome formatter overrides is genuinely useful. The troubleshooting section in `docs/development-guide-bmad-ui.md` covers the three most common onboarding blockers. pnpm commands are now consistent across all docs.

3. **Story 5.1's implementation was architecturally clean**  
   Node.js built-ins only (`node:fs`, `node:path`, `node:readline`) — no runtime dependencies. `npm pack --dry-run` confirmed that only `bin/install.mjs` and `_bmad-custom/bmad-ui/**` are included; all BMAD-specific directories excluded. Overwrite guard with readline prompt (default N) is the right UX. Local end-to-end tests passed.

4. **Code review for 5.2/5.3 caught a real regression**  
   After Story 5.3 updated `.vscode/settings.json` to configure Biome as default formatter, VS Code's built-in TypeScript formatter fired on save and corrupted three files (`package.json`, `vite.config.ts`, `styles.css`). Code review caught this and reverted before commit. The formatter conflict is exactly the kind of subtle issue that only appears in a review pass.

5. **pnpm vs npm ambiguity from Epic 4 retro was resolved correctly**  
   pnpm remains canonical for development (CI enforces `--frozen-lockfile`, CONTRIBUTING.md uses pnpm). For the npx install, the CLI installs into a target project using plain `npm install && npm run dev` next-steps — the right split. The package.json at repo root enables `npx bmad-method-ui install` from any machine without requiring pnpm.

## Key Insights and Learnings

1. **Sprint-status.yaml must be the single point of truth — story file status and sprint-status must always agree** — When they diverge (as with Story 5-1), any tooling or retrospective that reads sprint-status gets the wrong picture. Rule: changing a story file's `Status:` field and updating sprint-status.yaml must happen in the same commit.

2. **Action items die without story files — at this point this rule needs to be enforced at epic close** — The team has stated this principle in three consecutive retrospectives. The only enforcement mechanism is making it a hard gate: epic cannot be closed unless every action item from the previous retro either has a story file or is explicitly documented as accepted debt.

3. **"Verify and document" stories are high-value but easy to underscope** — Stories 5.2 and 5.3 were quick precisely because the underlying tooling was correct. If it hadn't been, they would have been much larger. The pattern: write these stories defensively with explicit "if broken, fix it" scope, not just verification.

4. **The formatter conflict in Story 5.3 is a known VS Code footgun — it should be in the troubleshooting section** — After updating `settings.json` to add Biome as default formatter, VS Code's built-in formatter can still fire if the Biome extension hasn't registered yet. This caused file corruption on save. The fix is already documented in `docs/development-guide-bmad-ui.md` — good.

5. **The npx install story (5.1) is technically complete but needs a publishing decision** — The CLI works. `npm pack --dry-run` passes. But AC #3 ("Given the npm package is published, When a user runs `npx bmad-method-ui install`...") requires actual publication. This needs a decision: npm registry under `bmad-method-ui`, GitHub Packages, or a different distribution strategy.
